# 工程项目全过程动态造价管理信息系统

本仓库是“工程管理信息系统”课程的统一项目基座。教师维护公共底座，学生按 GitHub Issue 领取任务，在独立分支完成模块后通过 Pull Request 提交验收。

当前网站与仓库检查结果见 `docs/site-audit-2026-07-13.md`。

- GitHub 仓库：https://github.com/dynamor2019/Project-Full-Process-Dynamic-Cost-Management-Information-System
- 在线预览：https://dynamor2019.github.io/Project-Full-Process-Dynamic-Cost-Management-Information-System/
- 学生协作规则：`docs/github-student-workflow.md`

## 项目目的

本项目的首要目的不是把学生培养成传统程序员，也不是追求功能庞大的商业软件，而是以一个统一的工程造价管理信息系统为载体，让缺少系统开发经验的工程造价学生学会使用 AI 编程工具，把真实的工程管理问题转化为可执行任务，并通过团队协作完成可运行、可验证、可交付的课程作品。

### 教学目的

学生通过本项目应完成以下能力转变：

1. 从“描述一个模糊想法”转变为能够说明用户、业务场景、输入数据、管理规则、输出结果和验收标准。
2. 从“让 AI 随意生成代码”转变为能够给 AI 分配边界明确的小任务，并检查其修改范围和结果。
3. 从“只会运行自己的页面”转变为能够在统一技术栈、统一数据结构和统一接口下与其他成员协作。
4. 从“看到页面能打开就认为完成”转变为能够使用专业依据、人工标准答案和测试用例验证结果。
5. 从“提交一次作业”转变为能够使用 GitHub Issue、分支、Commit、Pull Request 和 Release 管理完整交付过程。

### 专业目的

系统围绕同一个脱敏工程项目，把工程量清单、目标成本、计量支付、变更签证、合同风险等分散信息组织起来，形成“数据采集—规则处理—偏差识别—风险预警—管理决策—结果追踪”的工程管理闭环。

系统不是替代造价工程师作出最终判断，而是帮助其减少重复整理工作、及时发现成本和合同风险，并为人工复核与管理决策提供可追溯依据。

### 课程衔接目的

本仓库是三门课程的共同成果载体：

- 工程管理信息系统课程完成可运行的 `v0.1`，确保学生已经具备AI工具、工程任务、团队协作和系统交付的基本能力。
- 创新创业基础课程基于 `v0.1` 完成用户验证、价值分析和产品规划，形成 `v0.2-plan`。
- 创新创业实践课程继续开发、测试和试用，最终发布 `v1.0`。

即使三门课程间隔较长，代码、数据、任务、测试、讨论和未完成事项也通过同一GitHub仓库持续保存，后续课程不再重新选题、重新组建技术路线或从零开发。

### 成功判据

项目成功不以页面数量、代码行数或技术复杂度衡量，而以以下结果衡量：

- 每名学生都能独立配置环境、运行基座并完成至少一个可追踪任务。
- 每个小组都能提交一个输入、规则、输出完整闭环的工程造价管理模块。
- 关键计算和预警结果能够通过人工标准答案验证。
- 所有合格模块能够在统一项目中集成，`main` 分支保持可运行。
- 课程结束时能够发布带有代码、数据说明、测试报告、使用说明和贡献记录的正式版本。

## 项目基本情况

本项目面向工程造价专业学生，以同一个脱敏工程项目为业务背景，建设覆盖工程量清单、目标成本、计量支付、变更签证、合同风险和管理驾驶舱的 Web 信息系统。

项目采用前后端分离架构：浏览器端负责页面和交互，后端统一处理业务规则、数据校验和数据库访问。学生可以使用自己熟悉的本地 IDE 或 AI 编程 IDE，但不得自行更换课程技术栈、公共接口和数据库结构。

### 统一技术栈

| 层级 | 统一方案 | 课程约束 |
|---|---|---|
| 前端 | React + Vite + TypeScript | 统一使用 TypeScript，学生不得改动框架层 |
| 后端 | Node.js + Express | 提供 REST JSON API，业务规则放在后端 |
| 数据库 | SQLite | 使用本地单文件数据库，前端不得直接访问数据库 |
| SQLite 驱动 | `better-sqlite3` | 由后端统一封装数据库访问 |
| 包管理 | npm | 必须提交并使用 `package-lock.json` |
| 版本管理 | Git + GitHub | Issue、任务分支、Commit、Pull Request、Release |

推荐使用 Node.js 24 LTS，最低版本为 Node.js 22.12。Vite 当前要求 Node.js 20.19+ 或 22.12+，统一到受支持的 LTS 版本可以减少学生之间的环境差异。

SQLite 是无服务器、零配置的单文件数据库，适合本课程的本地开发和教学数据规模。仓库只提交数据库结构、迁移脚本和脱敏种子数据，不提交学生本机生成的 `.db`、`.sqlite` 或 `.sqlite3` 文件。

### 计划中的代码结构

```text
frontend/                    React + Vite + TypeScript 前端
  src/
    common/                  公共页面、组件、样式和请求封装
    modules/                 各造价业务模块页面
backend/                     Node.js + Express 后端
  src/
    common/                  配置、异常处理和数据库访问
    modules/                 各模块路由、服务和业务规则
database/
  schema.sql                 数据库结构
  seed.sql                   脱敏教学数据
  migrations/                数据结构变更记录
docs/                         项目章程、数据字典、规则和任务卡
tests/                        结构、接口和业务规则测试
```

所有前端模块通过后端 API 获取数据；禁止在 React 页面中直接写 SQL、保存数据库密码或复制一套独立业务规则。

前端统一使用全局 CSS 和统一 TypeScript 类型定义。学生后续只负责在既定业务模块内增加字段、表单项、读取、存储、修改、筛选和规则处理能力，不得改动公共导航、公共样式入口和框架层目录。

## 25 个任务池与统一索引

为支持两个班约 90 名学生按组协作开发，平台预置 25 个可认领任务主题，按 5 个生命周期阶段组织：

1. 前期策划与基础资料
2. 合同与清单管理
3. 目标成本与动态控制
4. 计量支付与过程结算
5. 变更签证与收尾分析

每组 4 人认领 1 个任务。允许部分任务未被选择，但每个任务都必须实现：

- 数据读取
- 数据新增
- 数据修改
- 数据查询
- 至少 1 条业务规则
- 至少 1 条统一索引登记

平台必须维护统一索引表 `form_index`。任何正式业务表单保存后，都要同步登记索引，用于按项目、合同、状态、业务类型和业务单号进行综合检索。

## 模块交付标准与成果列表

你不能把“页面做出来”当作最终完成。每个模块都必须形成“业务场景 -> 输入数据 -> 管理规则 -> 处理结果 -> 风险提示 -> 人工复核”的工程管理闭环。AI 可以辅助生成代码，但你必须能够说明业务逻辑、验算依据和工程管理意义。

### 模块通用 6 件套

你的小组认领任务后，无论属于哪个业务域，最终都必须提交以下 6 类成果：

1. 模块需求说明：说明使用者、业务场景、解决的问题和不做的范围。
2. 数据字段说明：列出字段名称、含义、类型、是否必填、样例值和数据来源。
3. 业务规则说明：写清计算公式、判断条件、风险阈值和人工复核方法。
4. 前端页面：提供录入、查询、展示和风险提示界面。
5. 后端接口：提供读取、新增、修改、查询和规则计算接口。
6. 验收材料：提交测试数据、预期结果、运行截图、PR 链接、已知问题和小组分工。

### 模块通用验收门槛

你的模块只有同时满足以下条件，才可以提交验收：

- 至少有 1 个真实工程造价或工程管理业务场景，不允许只做普通增删改查。
- 至少有 5 条正常样例数据和 2 条异常样例数据。
- 至少实现数据读取、数据新增、数据修改和条件查询。
- 至少实现 1 条核心计算规则，例如偏差率、累计比例、超期天数、风险等级。
- 至少实现 1 条风险预警规则，并能在页面中明确显示预警结果。
- 正式业务数据保存后，必须同步写入或更新统一索引 `form_index`。
- 前端页面必须通过后端 API 获取数据，禁止前端直接访问 SQLite。
- 模块结果必须能够用人工标准答案复核，不能只以 AI 生成结果作为依据。
- Pull Request 必须填写测试过程、截图、影响范围和已知问题。
- 你能在现场讲清楚输入、规则、输出、风险提示和工程管理价值。

### 弹性交付层级

为避免任务过大或过小，每个模块按三层设计。你先完成必达层，再根据进度继续扩展。

| 层级 | 作用 | 交付要求 | 适用情况 |
|---|---|---|---|
| 必达层 | 保证你有可运行成果 | 完成录入、查询、修改、保存、1 条业务规则、1 条预警规则和验收截图 | 所有小组必须完成 |
| 标准层 | 达到课程预期质量 | 增加分类统计、偏差计算、风险清单、人工验算表和模块说明 | 正常进度小组完成 |
| 挑战层 | 给进度较快的小组继续扩展 | 增加趋势图、跨模块联动、自动报告、AI 辅助风险说明和更完整测试 | 进度较快小组选做 |

如果你的小组进度偏慢，先把必达层闭环做扎实，不临时扩大范围；如果你的小组进度很快，不要继续堆普通页面，而要提高规则准确性、测试完整性、跨模块联动和工程管理解释质量。

### 诊断任务与节奏调整

在模块正式开发前，你需要先完成一次短诊断任务，例如“支付金额超比例预警小功能”：

1. 增加付款比例字段。
2. 后端计算累计付款比例。
3. 超过合同价 80% 显示黄色预警。
4. 超过合同价 90% 显示红色预警。
5. 提交 Pull Request。
6. 现场解释计算逻辑和测试数据。

根据全班诊断任务完成情况，后续难度会这样调整：

| 完成情况 | 后续策略 |
|---|---|
| 70% 以上学生能按时完成 | 正常推进标准层，并开放挑战层 |
| 40% 至 70% 学生能按时完成 | 保留标准层，减少挑战层和跨模块联动 |
| 低于 40% 学生能按时完成 | 缩小模块，只要求必达层闭环 |

## 学生自配 IDE 与环境

课程允许学生使用 VS Code、TRAE、CodeBuddy 或其他具备终端和 Git 能力的 IDE。教师不负责适配不同 IDE 的按钮和插件，只统一支持仓库命令、目录规范和验收结果。

### 必装环境

1. Git。
2. Node.js 24 LTS；最低不得低于 22.12。
3. Node.js 自带的 npm。
4. 任意支持终端、Git 和 JavaScript 的 IDE。
5. Chrome、Edge 或其他现代浏览器。

安装后在 IDE 终端执行：

```bash
git --version
node --version
npm --version
```

三条命令均能显示版本号，才算完成基础环境配置。不得要求学生全局安装项目依赖；项目依赖统一写入 `package.json` 并通过 `npm install` 安装。

本仓库在当前维护机上优先复用已安装的 Node.js 和现有前端依赖，不再额外下载新的依赖包；学生克隆到自己的电脑后，仍应按 README 自行执行 `npm install` 获取所需依赖。

### 基座完成后的统一命令

后续公共底座应在仓库根目录提供以下命令，学生无须记忆不同子项目的启动方式：

```bash
npm install        # 安装前后端依赖
npm run setup      # 初始化本地 SQLite 数据库
npm run dev        # 同时启动前端和后端
npm run test       # 执行自动测试
npm run check      # 执行格式、测试和构建检查
```

约定开发地址：

- 前端：`http://localhost:5173`
- 后端：`http://localhost:3000`
- 健康检查：`http://localhost:3000/api/health`

上述命令和地址将在 React、Node.js 与 SQLite 公共底座实现后启用；当前结构版本尚未提供可运行的 Web 应用。

### 环境验收任务

你必须独立完成一次 `ENV-CHECK`：

1. Clone 教师仓库。
2. 在自己的 IDE 中打开项目。
3. 在自己的克隆环境中安装依赖并启动前后端。
4. 浏览器打开首页和后端健康检查接口。
5. 初始化本地 SQLite 数据库并读取一条教师示例数据。

环境验收未通过的学生不能进入模块开发。

### 使用 AI IDE 的边界

- 开始任务前，先让 AI 阅读 `README.md`、对应 Issue、数据字典和业务规则。
- 要求 AI 先说明修改计划和文件范围，再允许其修改。
- 禁止让 AI 更换 React、Express、SQLite 或 npm。
- 禁止让 AI 删除测试、绕过校验或把密钥写入代码。
- 你必须运行测试，并解释输入、规则、输出和人工复核点。

## 使用边界

- 不提交真实涉密工程数据、个人信息、`.env` 或 API 密钥。
- 未经批准不修改 `src/common/`、公共数据结构和整体技术栈。
- 每个任务原则上控制在 1 至 2 学时，超出时必须继续拆分。

## 下一阶段建设任务

下一阶段只建设“所有学生都能运行的公共技术底座”，暂不开发清单、成本、支付、变更、合同风险等正式业务模块。

### 实施顺序

1. 创建 `frontend/`，使用 React + Vite + JavaScript。
2. 创建 `backend/`，使用 Node.js + Express，统一使用 ES Modules。
3. 创建 `database/schema.sql`、`database/seed.sql` 和迁移目录。
4. 在后端封装 SQLite 连接，不允许前端直接访问数据库。
5. 增加 `GET /api/health` 健康检查接口。
6. 增加一个“课程基座状态”前端页面，显示前端、后端和数据库状态。
7. 在根目录建立统一 `package.json` 和 npm scripts。
8. 实现 `npm install`、`npm run setup`、`npm run dev`、`npm run test`、`npm run check`。
9. 更新 `start.bat`，使其能够启动实际应用而不只是检查目录。
10. 更新 GitHub Actions，执行安装、测试和构建检查。

### 公共底座验收标准

只有同时满足以下条件，才可以宣布公共底座完成：

- 从干净 Clone 开始，不依赖开发者电脑中的全局项目包。
- Node.js 24 LTS 或不低于 22.12 的版本可以运行。
- 根目录执行 `npm install` 成功。
- 执行 `npm run setup` 能创建本地 SQLite 数据库并写入脱敏种子数据。
- 执行 `npm run dev` 能同时启动前端和后端。
- `http://localhost:5173` 能显示基座状态页面。
- `http://localhost:3000/api/health` 返回成功状态，并证明数据库可连接。
- `npm run test` 和 `npm run check` 全部通过。
- 本地数据库、`.env`、密钥和 `node_modules` 不进入 Git。
- README 中的命令与实际命令完全一致。
